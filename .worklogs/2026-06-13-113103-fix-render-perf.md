# Fix UI freezing while adjusting settings (render-loop perf)

**Date:** 2026-06-13 11:31
**Scope:** src/render/canvas.ts

## Summary
Dragging sliders froze the UI. Measured the render and found the per-glyph draw loop was the entire cost. Fixed by (1) grouping glyphs by a quantized font string so ctx.font is parsed once per distinct size/weight instead of once per glyph, and (2) rendering rotated glyphs with setTransform instead of save/translate/rotate/restore. Default cellPx steady-state dropped from ~44 ms/frame to ~6.6 ms; the extreme cellPx=7 (16k glyphs) roughly halved.

## Context & Problem
Slider input fires rapidly; each change is coalesced through one requestAnimationFrame, so at most one draw per frame. The freeze was therefore per-frame draw cost, not event flooding.

Instrumented draw() (lum / layout / draw timings, glyph count) and stress-tested in headless chromium:
- Default cellPx=13: ~4585 glyphs, lum 0.7 ms, layout 5 ms, draw 44 ms.
- cellPx=7: ~15971 glyphs, draw 212 ms.
- Gamma drag at cellPx=7 (original code): avg 326 ms/frame.

So luminance and layout were negligible; the glyph draw loop was ~90% of the time. Caching luminance was therefore pointless and was not done.

Isolation experiments:
- Setting ctx.font only when the (quantized) string changed, in row-major order, did NOT help (262 ms): adjacent cells rarely share a quantized font, so font was still set ~16k times.
- Grouping by font (one font-set per distinct string): cellPx=7 draw 160 ms, cellPx=13 draw 12 ms. Big win at low glyph counts; proved ctx.font reparse was a major cost.
- Disabling rotation (warp=0) with grouping: cellPx=7 draw 54 ms. Proved save/translate/rotate/restore was ~3x the cost of plain fillText.

## Decisions Made

### Group glyphs by quantized font string
- **Chose:** Map<fontString, glyphs[]>; fontSize rounded to 0.5 px, weight to nearest 10; set ctx.font once per group.
- **Why:** ctx.font assignment reparses the font every call; with continuous weight/size every glyph had a unique string, so it was parsed ~16k times/frame. Grouping cuts that to a few hundred.
- **Safety of reordering:** all marks are the same ink color (#15151a). Source-over compositing of same-color layers is order-independent (C[a + b(1-a)] + bg(1-a)(1-b) is symmetric in a,b), so regrouping cannot change the rendered pixels. Verified visually (output identical).
- **Quantization is imperceptible and deterministic:** 0.5 px / 10-weight buckets are not visible at these sizes, and being deterministic it preserves the preview-toggle e2e (exact hash restore).
- **Alternatives considered:** in-order font-change skipping (rejected - did not reduce font sets); caching luminance (rejected - <3 ms, irrelevant).

### setTransform for rotated glyphs
- **Chose:** ctx.setTransform(cos, sin, -sin, cos, x, y) + fillText(ch, 0, 0); reset to identity only when switching back to non-rotated, identity tracked with a flag.
- **Why:** save/translate/rotate/restore is 4 calls plus a full state-stack snapshot per glyph; setTransform is one matrix write. translate(x,y);rotate(r) is exactly setTransform(cos,sin,-sin,cos,x,y), so output is identical.

## Results
- Default cellPx=13 gamma drag: avg draw 6.6 ms, max 14 ms (was 44 ms) - smooth.
- cellPx=7 warm gamma drag: avg ~152 ms (was ~326 ms) - halved; still heavy because 16k fillText calls are an irreducible floor at the minimum cell size.
- 31 unit tests + 7 e2e green; typecheck + build green; render visually unchanged.

## Architectural Notes
Draw loop extracted into private drawGlyphs(ctx, glyphs). The plot-preview path (drawPlotPreview) is unchanged and still returns early; it was not the reported bottleneck.
